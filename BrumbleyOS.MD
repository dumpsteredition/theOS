# BrumbleyOS

For visual and interaction design implementation, BrumbleyOS_Design_Language.md is the canonical source of truth. The approved homepage, navigation, top bar, and footer are locked reference implementations.

## Project Concept

**BrumbleyOS** is Kyle Brumbley’s personal website reimagined as a premium, interactive software experience.

It should **look, feel, and behave like a modern application**, but the copy should **read like a polished personal website**.

The visitor should feel like they have opened a high-end product surface that lets them explore Kyle’s work, thinking, product instincts, UX taste, case files, operating defaults, and ways to connect.

This is not a normal portfolio. It is a personal product interface.

Note: Settings and Files were removed from the primary public site experience. Remaining references in this document are archival concept notes unless a future prompt explicitly restores them.

## Current North Star

> **App surface. Website voice.**

BrumbleyOS should look and quack like software, but it should not talk like fake software.

The application feeling should come from:

- the layout
- the navigation model
- the shell
- the interactions
- the animations
- the command palette
- the visualizations
- the embedded profile/feed window
- the case-file explorer
- the settings-style operating defaults
- the tactile controls and app states

The content should not rely on fake system language to sell the concept.

The copy should sound like a strong personal website for a product/UX/healthcare AI leader:

- clear
- human
- sharp
- outcome-oriented
- credible
- low-fluff
- direct
- slightly witty when it earns the moment

## Content Strategy: App Surface, Website Voice

### The rule

The site should **feel like software through design and interaction**, not through over-literal app-roleplay copy.

Do not make every label sound like a fake security console, fake operating system, or sci-fi dashboard.

The interface can be playful and app-like. The words should be mature and useful.

### Translation

- **Visual layer:** application
- **Interaction layer:** application
- **Layout model:** application
- **Navigation model:** application
- **Animation/state model:** application
- **Content/copy layer:** personal website / landing page

### What changed from the original concept

The original framing leaned heavily on phrases like:

- “Logged in as Kyle”
- “Read-only session”
- “Session Status”
- “Access granted”
- “Workspace Console”
- “Operator Mode”

That helped establish the idea early, but it is now too literal. It risks making the site feel gimmicky.

Going forward, BrumbleyOS should keep the feeling of being inside Kyle’s personal product environment, but the main copy should be more natural, credible, and website-appropriate.

### Usage guidance

Use app-like words when they are attached to actual app-like surfaces or controls:

- navigation
- command menu
- settings
- case files
- system logs
- operating defaults
- profile
- launcher
- filters
- charts
- panels

Avoid using app-like words as the main marketing message.

## Experience Principles

### 1. Premium before gimmicky

The app metaphor should feel high-end, not like a joke site.

It should borrow from modern product tools and SaaS interfaces while keeping the content credible and human.

### 2. Let interaction carry the concept

The app feeling should come from:

- app shell
- hover states
- command palette
- panels
- embedded windows
- visualizations
- launchers
- animated transitions
- drawers/modals
- tactile controls

Not from repeating “session,” “access,” or “logged in” everywhere.

### 3. Every playful interaction must reveal real substance

No empty gimmicks.

No fake feature should exist only to be cute.

Every playful interaction should communicate something real about Kyle’s thinking, working style, or product taste.

### 4. Make the visitor feel like they discovered something

The site should feel explorable.

It should reward curiosity.

Small hidden moments are good, but the core content must remain obvious.

### 5. Readable and usable first

Recruiters, collaborators, founders, clients, and peers should quickly understand Kyle’s strengths.

The app concept should not get in the way of comprehension.

### 6. Desktop-first, mobile-responsible

The richest version should shine on desktop.

Mobile should still work cleanly and not collapse into a broken novelty.

## Signature Interaction: UX Bullshit Detector

This is one of the strongest personality moments so far.

### Setting

**UX Bullshit Detector**  
Status: **Always On**

### Visual Behavior

The toggle should look switched on, but slightly broken from overuse.

Details:

- toggle is visibly on
- active color/glow flickers slightly like a broken light
- knob is tilted or hanging 2–3px off-axis
- subtle damaged/misaligned feel
- funny but still polished

### Hover Behavior

Tooltip idea:

> Cannot be disabled. Attempts have failed.

### Click Behavior

When the user tries to turn it off:

1. toggle briefly tries to move off
2. glow flickers
3. knob snaps back to on
4. toast appears

Toast copy:

> **Protected setting**  
> UX Bullshit Detector cannot be disabled.

### Restraint Rule

This works because it is one great joke among mostly polished, credible settings.

Do not make every setting goofy.

Use maybe 2–3 personality-rich settings max:

- UX Bullshit Detector: Always On, broken toggle
- Fluff Filter: Aggressive
- Meeting Theater Blocker: Enabled

Everything else should be credible and sharp.
